

# 软件测试的定义：
- 1. 在规定的条件下对程序进行操作，以发现程序错误，衡量软件质量，并对其是否能满足设计要求进行评估的过程
- 2. 是使用人工或自动的手段来运行或测定某个软件的过程，在于检测它是否满足规定的需求或弄清预期结果与实际结果之间的差别。

# 软件测试的常见术语：
     
- 1. c/s: c(client)是指客户端，s(server)是指服务器端，是基于局域网或互联网的，需要一台服务器来安装服务器端软件
- 2. b/s: b（browser）是浏览器,s(server)是指服务器,与c/s结构软件的区别在于，不需要安装客户端，只需要有浏览器，就可以直接使用，

# 软件缺陷

> 是一种泛指、可以指功能的错误，性能低下，易用性差



# 什么是软件缺陷
- 1. 软件未达到产品说明书表明功能
- 2. 软件出现了说明书内不会出现的错误
- 3. 软件功能超出产品说明书的范围
- 4. 软件未达到产品说明书的目标
- 5. 测试人员认为难以理解、不易使用、运行速度慢、用户认为不好用 
  

 #  缺陷产生的原因：
 
 > 工期短，任务大；需求不断变化；软件复杂性；软硬件支持不完善；沟通交流不够；文档不完善；程序设计错误


 # 缺陷报告

 ## bug重现

 - 1. 不要想当然的接受任何假设，要做好记录
 - 2. 查找时间依赖和竞争条件的问题
 - 3. 边界条件软件缺陷，内存泄漏和数据溢出等白盒问题会显露出来
 - 4. 状态缺陷仅在特定软件状态中显露
 - 5. 考虑资源依赖性和内存、网络、硬件共享的相互作用


## 无法重现的bug

- 1. 做好缺陷记录，并提交出去
- 2. 对于难以再现的缺陷要合理安排时间，确保项目的进度
- 3. 最后在测试中对为在现的缺陷予以关注


## 缺陷报告

>是对缺陷进行记录、分类和跟踪文档；提供准确、完整、简洁、一致的缺陷报告是体现软件测试的专业性、高质量的主要指标


## 缺陷报告包含信息

## 缺陷报告的写作准则（5c）


- 1. Correct（准确）：每个组成部分的描述准确，不会引起误解
- 2. Clear(清晰)：每个组成部分的描述清晰，易于理解
- 3. Concise(简洁)：只包含必不可少的信息，不包括任何多余的内容
- 4. complete(完整)：包含复现该缺陷的完整步骤和其他本质信息
- 5. Consistent（一致）：按照一致的格式书写全部缺陷报告


## 缺陷报告的组织架构


- 1. 缺陷的标题（按缺陷发生的原因与结果的方式书写；避免使用迷糊不清的词语；关键字；）
- 2. 缺陷的基本信息
- 3. 测试的软件和硬件环境
- 4. 测试的软件版本
- 5. 缺陷的类型
- 6. 缺陷的严重程度
- 7. 缺陷的处理优先级
- 8. 复现缺陷的操作步骤
- 9. 缺陷的实际结果描述
- 10. 期望的正确结果描述
- 11. 注释文字和截图的缺陷图像


## 缺陷报告原则

- 1. 组织Structure：记录好缺陷报告给开发人员
- 2. 重现Reproduce:在写缺陷报告之前必须在检查问题是否重现（至少3次）
- 3. 隔离Lsolate:隔离错误。改变一些变量的方法，如系统配置，它可能可以改变错误的状态。
- 4. 归纳Generalize:发现一个已隔离的，可重现的问题后，应该对问题进行归纳。
- 5. 对比Compare:回归测试中发现有错误时，要与之前测试的数据进行对比，找出错误的原因
- 6. 总结Summarize:缺陷报告第一行写上错误总结很关键。
- 7. 精简Condense:
- 8. 消除歧义Disambiguate 
- 9. 中立性
- 10. 检查Review:

# 禅道项目管理系统

## 软件介绍

> 是一款基于Scrum（敏捷）思想并集产品管理、项目管理、测似管理于一体，同时包括了事务发管理、组织管理、等项目管理软件。

## 禅道软件用户角色
- 1. 系统管理员（Admin）主要负责添加永华、分配权力
- 2. 产品人员（product owner）产品人员主要负责产品管理
- 3. 项目经理（project manager）通过项目协调产品人员、开发人员，测试人员完成产品
- 4. 开发人员（developer）开发人员负责产品的研发
- 5. 测试人员（QA）保证产品的质量

# 禅道项目模式基本流程：

- 1. 产品经理创建产品
- 2. 产品经理创建需求
- 3. 项目经理创建项目
- 4. 项目经理确定项目要做的需求
- 5. 项目经理分解任务，指派到人
- 6. 测试人员测试，提交bug
